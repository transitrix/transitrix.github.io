Install & start

Install Transitrix Studio in your editor, then author your first diagram — a Goals tree rendered from plain text, in a few minutes.

What are you doing?

  • Just want the viewers — install Studio and the Mermaid extension.
  • Starting fresh in an empty folder — author your first diagram.
    • Fast path — the onboarding Skill
    • Manual path — no Claude Code required
  • Setting up your organisation's Transitrix repo — the real thing: zones, manifest, agent guides, CI.
  • Joining a repository your team already has — clone, render, and contribute.

Install the viewers

Transitrix diagrams are plain text. Two editor extensions render them live as you type — together they cover every layer of one model, from the board to the developer.

Transitrix Studio — the business-facing layers

Pick your channel; the extension is identical across all of them.

  • VS Code — the VS Code Marketplace. Or search Transitrix Studio in the Extensions panel.
  • Cursor, VSCodium, Windsurf — the Open VSX Registry. Same extension and identifier; search Transitrix Studio in the Extensions panel.
  • IntelliJ IDEA — the JetBrains Marketplace. Also installs into the other JetBrains IDEs.

Markdown Preview Mermaid Support — the technical layers

Studio renders the business-facing layers; the technical layers are standard Mermaid. Install Markdown Preview Mermaid Support alongside Studio (on the VS Code Marketplace and Open VSX) to render sequence, state, ER and C4 component diagrams, down to the application and technology layers.

One model, every layer. Studio renders the business-facing layers with purpose-built notations that go beyond generic diagramming — its BPMN, process and activity views, not stripped-down Mermaid copies. Mermaid then renders the technical layers Transitrix doesn't specialise — the same plain-text model, every layer, from the board to the developer. And nothing is locked in: Transitrix is built on open standards (ArchiMate 3.2, BPMN 2.0), the technical views are standard Mermaid that renders in any tool, and your files stay diffable text on an open, MIT-licensed methodology — readable with or without our tooling.

Your first diagram

With the viewers installed, author your first Transitrix artefact: a Goals tree rendered from a .goals.transitrix.yaml file. Two paths.

The fast path — onboarding Skill

Requires Claude Code with access to the Claude marketplace. The Skill scaffolds a repository — or orients you in one your organisation already has — and authors your first file. In a Claude Code session:

/plugin marketplace add transitrix/methodology
/plugin install transitrix@transitrix-methodology
/transitrix:onboard

The Skill asks what you want to model first, scaffolds the repo layout (canon/, field/, codex/), and authors a starter Goals tree file. Open it in VS Code — the diagram renders alongside as you type. View the Skill on GitHub →

The manual path

No Claude Code required — and no public GitHub access needed.

  1. Open VS Code in a new empty folder (Studio installed — see above).
  2. Create the file views/goals/HELLO.goals.transitrix.yaml.
  3. Paste this into it: 
    notation: goals
spec_version: "0.1"

id: GOALS-HELLO-1
name: "Hello — Goals Tree"
description: "A starter Goals tree."
period: "2026"
version: "0.1"
date: "2026-01-01"
author: Me

goal_types:
  - { name: "Strategic Goal", level: 0 }
  - { name: "Tactical Goal",  level: 1 }

goals:
  - id: GOAL-1
    name: "Expand into new markets"
    type: "Strategic Goal"
    level: 0

  - id: GOAL-1-1
    name: "Enter the German market by Q3"
    type: "Tactical Goal"
    level: 1
    parent: GOAL-1

  - id: GOAL-1-2
    name: "Sign two distribution partnerships"
    type: "Tactical Goal"
    level: 1
    parent: GOAL-1

  - id: GOAL-2
    name: "Improve customer retention"
    type: "Strategic Goal"
    level: 0

  - id: GOAL-2-1
    name: "Reduce churn by 20% in H2"
    type: "Tactical Goal"
    level: 1
    parent: GOAL-2
  4. Studio renders the Goals tree in the preview panel. Edit a goal name — the diagram updates live.

To validate from the command line (requires npm): npx @transitrix/cli validate views/goals/HELLO.goals.transitrix.yaml

Setting up your organisation's Transitrix repository

Not one loose diagram file — the real thing. A zoned repository with canon/ (validated model), field/ (raw inputs), and codex/ (laws, policies, standards); a transitrix.yaml manifest declaring which notations and zones you use; an assistant-neutral AGENTS.md plus ANALYST.md and VALIDATOR.md role guides; a whole-repo validator under .validators/; and a GitHub Actions workflow that gates every pull request on structural and referential integrity.

Copy this prompt, paste it into a chat with your coding agent — Claude Code, Cursor, Copilot Chat, Gemini CLI, anything that can fetch a URL and follow instructions. From the parent folder of the new repo:

Fetch and read
https://raw.githubusercontent.com/transitrix/methodology/main/transitrix/skills/onboard/SKILL.md
then follow it exactly, with one substitution: wherever it references a
file under templates/ (whether written as the bare path or via
${CLAUDE_SKILL_DIR}/templates/<file>), fetch it instead from
https://raw.githubusercontent.com/transitrix/methodology/main/transitrix/skills/onboard/templates/<file>

The agent asks what you want to model first, scaffolds the zoned layout, drops in AGENTS.md, ANALYST.md, VALIDATOR.md, the transitrix.yaml manifest, and VS Code / Copilot pointers, fetches the whole-repo validator and CI workflow from a pinned methodology release, and then authors a first notation file (Goals, DGCA, capability map — your choice) so a diagram renders in Studio the moment you open it. The Skill in the methodology repo stays the single source of truth — this prompt just points your agent at it. View the Skill on GitHub →

Already use the Claude Code plugin workflow?

If your surface supports /plugin commands — the standalone terminal claude CLI does; the VS Code extension's chat surface does not yet — you can install the Skill instead of pasting the prompt above. Same output, from the parent folder of the new repo:

/plugin marketplace add transitrix/methodology
/plugin install transitrix@transitrix-methodology
/transitrix:onboard

Joining a repository your organisation already has

Someone on your team has already set up the model. You don't scaffold anything — you clone it, watch it render, and make your first change like any other contribution.

  1. Clone the repository and open it in VS Code (both viewers installed — see above): 
    git clone <your-org-repo-url>
  2. Open any .transitrix.yaml file under canon/ or views/. Studio renders your organisation's real diagram beside it.
  3. Read AGENTS.md at the repository root — the orientation guide the repo ships with: where each zone lives (canon/, field/, codex/), which notations are in use, and the local conventions.
  4. Make a change — edit an element or add one — then validate it: 
    npx @transitrix/cli validate <file>
  5. Open a pull request. Architecture changes are reviewed like code, and the repository's CI re-runs validation on every PR.

That's the whole loop — clone, render, change, validate, PR — the same review-as-code workflow your team already uses for source.
