Schnellstart

Ihr erstes Transitrix-Artefakt — ein aus einer YAML-Datei gerenderter Zielbaum — in wenigen Minuten.

Zwei Einstiege. Neu anfangen — ein neues Repository für Ihre Organisation anlegen. Oder beitreten — Ihre Organisation hat bereits ein Transitrix-Repository und Sie möchten mitarbeiten. In beiden Fällen installieren Sie zuerst die beiden Viewer.

Die Viewer installieren

Transitrix-Diagramme sind reiner Text. Zwei Editor-Erweiterungen rendern sie live beim Tippen — zusammen decken sie jede Ebene eines einzigen Modells ab, vom Vorstand bis zur Entwicklung:

  • Transitrix Studio — die Enterprise-Notationen für die geschäftsnahen Ebenen: Zielbäume, FGCA-/FGA-Strategieketten, Fähigkeitskarten, Prozess-Blueprints, BPMN und mehr. Für VS Code, Cursor, VSCodium und Windsurf (suchen Sie Transitrix Studio im Erweiterungen-Panel), dazu ein Plugin für IntelliJ IDEA im JetBrains-Marketplace.
  • Markdown Preview Mermaid Support — rendert die technischen Ebenen in standardmäßigem Mermaid: Sequenz-, Zustands-, ER- und C4-Komponentendiagramme, bis hinunter zur Anwendungs- und Technologieebene. Im VS-Code-Marketplace und auf Open VSX.

Ein Modell, jede Ebene. Studio rendert die geschäftsnahen Ebenen mit eigens entwickelten Notationen, die über generisches Diagrammieren hinausgehen — seine BPMN-, Prozess- und Aktivitätsansichten, keine abgespeckten Mermaid-Kopien. Mermaid rendert dann die technischen Ebenen, auf die Transitrix nicht spezialisiert ist — beide aus demselben reinen Textmodell, sodass eine einzige Quelle Vorstand und Entwicklung gleichermaßen dient. Und nichts bindet Sie: Transitrix baut auf offenen Standards auf (ArchiMate 3.2, BPMN 2.0), die technischen Ansichten sind standardmäßiges Mermaid, das in jedem Werkzeug rendert, und die Dateien bleiben diff-barer Text auf einer offenen, MIT-lizenzierten Methodik — lesbar mit oder ohne uns.

Neu anfangen — ein neues Repository

Legen Sie ein Repo an und erstellen Sie Ihr erstes Artefakt: einen Zielbaum, gerendert aus einer .goals.transitrix.yaml-Datei. Zwei Wege.

Der schnelle Weg — Onboarding-Skill

Erfordert Claude Code mit Zugriff auf den Claude-Marketplace. In einer Claude-Code-Sitzung:

/plugin marketplace add transitrix/methodology
/plugin install transitrix@transitrix-methodology
/transitrix:onboard

Der Skill fragt, was Sie zuerst modellieren möchten, legt die Repo-Struktur an (canon/, field/, codex/) und erstellt eine erste Zielbaumdatei. Öffnen Sie diese in VS Code — das Diagramm wird beim Tippen daneben gerendert.

Der manuelle Weg

Kein Claude Code erforderlich — und kein Zugriff auf öffentliches GitHub nötig.

  1. Öffnen Sie VS Code in einem neuen leeren Ordner (Studio installiert — siehe oben).
  2. Erstellen Sie die Datei views/goals/HELLO.goals.transitrix.yaml.
  3. Fügen Sie Folgendes ein: 
    notation: goals
spec_version: "0.1"

id: GOALS-HELLO-1
name: "Hello — Goals Tree"
description: "A starter Goals tree."
period: "2026"
version: "0.1"
date: "2026-01-01"
author: Me

goal_types:
  - { name: "Strategic Goal", level: 0 }
  - { name: "Tactical Goal",  level: 1 }

goals:
  - id: GOAL-1
    name: "Expand into new markets"
    type: "Strategic Goal"
    level: 0

  - id: GOAL-1-1
    name: "Enter the German market by Q3"
    type: "Tactical Goal"
    level: 1
    parent: GOAL-1

  - id: GOAL-1-2
    name: "Sign two distribution partnerships"
    type: "Tactical Goal"
    level: 1
    parent: GOAL-1

  - id: GOAL-2
    name: "Improve customer retention"
    type: "Strategic Goal"
    level: 0

  - id: GOAL-2-1
    name: "Reduce churn by 20% in H2"
    type: "Tactical Goal"
    level: 1
    parent: GOAL-2
  4. Studio rendert den Zielbaum im Vorschaufenster. Bearbeiten Sie einen Zielnamen — das Diagramm aktualisiert sich in Echtzeit.

Zur Validierung über die Kommandozeile (erfordert npm): npx @transitrix/cli validate views/goals/HELLO.goals.transitrix.yaml

Einem bestehenden Repository Ihrer Organisation beitreten

Jemand in Ihrem Team hat das Modell bereits aufgesetzt. Sie legen nichts an — Sie klonen es, sehen es rendern und machen Ihre erste Änderung wie jeden anderen Beitrag.

  1. Klonen Sie das Repository und öffnen Sie es in VS Code (beide Viewer installiert — siehe oben): 
    git clone <repo-url-ihrer-organisation>
  2. Öffnen Sie eine beliebige .transitrix.yaml-Datei unter canon/ oder views/. Studio rendert das echte Diagramm Ihrer Organisation daneben.
  3. Lesen Sie AGENTS.md im Repository-Stammverzeichnis — der Orientierungsleitfaden, den das Repo mitbringt: wo jede Zone liegt (canon/, field/, codex/), welche Notationen verwendet werden und die lokalen Konventionen.
  4. Nehmen Sie eine Änderung vor — ein Element bearbeiten oder hinzufügen — und validieren Sie sie: 
    npx @transitrix/cli validate <datei>
  5. Öffnen Sie einen Pull Request. Architekturänderungen werden wie Code geprüft, und die CI des Repositorys führt die Validierung bei jedem PR erneut aus.

Das ist die ganze Schleife — klonen, rendern, ändern, validieren, PR — derselbe Review-as-Code-Workflow, den Ihr Team bereits für Quellcode nutzt.
